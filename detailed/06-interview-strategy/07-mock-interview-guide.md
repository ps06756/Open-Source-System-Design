# Mock Interview Guide

How to simulate and conduct realistic system design practice sessions.

---

## Why Mock Interviews

```
┌─────────────────────────────────────────────────────────────┐
│              The Value of Mock Interviews                   │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Solo practice limitations:                                 │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  • No time pressure                                    │ │
│  │  • No unexpected questions                             │ │
│  │  • Can look up answers                                 │ │
│  │  • No communication practice                           │ │
│  │  • No feedback on blind spots                          │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  Mock interview benefits:                                   │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  • Real-time pressure simulation                       │ │
│  │  • Practice verbal communication                       │ │
│  │  • Learn to handle challenges                          │ │
│  │  • Discover gaps in knowledge                          │ │
│  │  • Build confidence                                    │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  Recommended: 2-3 mock interviews before real ones         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## Mock Interview Structure

```
┌─────────────────────────────────────────────────────────────┐
│               Interview Structure                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Total Time: 60 minutes                                     │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  0:00 - 0:02  Introduction (2 min)                     │ │
│  │  ────────────────────────────────                       │ │
│  │  • Interviewer introduces problem                      │ │
│  │  • Keep it brief, like real interview                  │ │
│  │                                                         │ │
│  │  0:02 - 0:47  Design Session (45 min)                  │ │
│  │  ────────────────────────────────                       │ │
│  │  • Candidate drives the discussion                     │ │
│  │  • Interviewer asks follow-ups                         │ │
│  │  • Use whiteboard/paper                                │ │
│  │                                                         │ │
│  │  0:47 - 0:50  Candidate Questions (3 min)              │ │
│  │  ────────────────────────────────                       │ │
│  │  • Candidate asks about role/company                   │ │
│  │  • Practice this part too!                             │ │
│  │                                                         │ │
│  │  0:50 - 1:00  Feedback Session (10 min)                │ │
│  │  ────────────────────────────────                       │ │
│  │  • Detailed feedback from interviewer                  │ │
│  │  • Discussion of alternative approaches                │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## For the Interviewer

### How to Give the Problem

```
┌─────────────────────────────────────────────────────────────┐
│              Interviewer Guide: Problem Setup               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Opening Statement:                                         │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  "Today we're going to design [SYSTEM]. I'll give you  │ │
│  │   a brief description, and then I'd like you to drive  │ │
│  │   the discussion. Feel free to ask clarifying          │ │
│  │   questions. Ready?"                                   │ │
│  │                                                         │ │
│  │  [Describe the system in 2-3 sentences]                │ │
│  │                                                         │ │
│  │  "Take a moment to gather your thoughts, and let me    │ │
│  │   know when you're ready to begin."                    │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  Don't:                                                     │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  ❌ Give too much detail upfront                       │ │
│  │  ❌ Answer questions before they're asked              │ │
│  │  ❌ Tell them exactly what to design                   │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  Do:                                                        │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  ✓ Be vague enough to require clarification           │ │
│  │  ✓ Have scale numbers ready when asked                │ │
│  │  ✓ Let them lead the conversation                     │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Questions to Ask

```
┌─────────────────────────────────────────────────────────────┐
│            Interviewer Guide: Follow-up Questions           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  At Requirements Phase:                                     │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  If they don't ask about scale:                        │ │
│  │  "What scale are you designing for?"                   │ │
│  │                                                         │ │
│  │  If they take on too much scope:                       │ │
│  │  "That's a lot. Which features should we prioritize?"  │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  During Design:                                             │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  Testing depth:                                         │ │
│  │  "Why did you choose [X] over [Y]?"                    │ │
│  │  "How does [component] work internally?"               │ │
│  │                                                         │ │
│  │  Finding edge cases:                                    │ │
│  │  "What happens if [component] fails?"                  │ │
│  │  "How would you handle [extreme case]?"                │ │
│  │                                                         │ │
│  │  Exploring alternatives:                                │ │
│  │  "Have you considered [alternative approach]?"         │ │
│  │  "What are the trade-offs of your approach?"           │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  If They're Stuck:                                          │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  Give hints progressively:                             │ │
│  │  1. "Think about [general area]..."                    │ │
│  │  2. "What about using [specific technique]?"           │ │
│  │  3. "In practice, companies often [solution]..."       │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  If They're Off Track:                                      │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  Redirect gently:                                       │ │
│  │  "That's interesting, but let's focus on [X]..."       │ │
│  │  "Before we go deeper there, can you show me [Y]?"     │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Giving Feedback

```
┌─────────────────────────────────────────────────────────────┐
│              Interviewer Guide: Feedback                    │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Structure your feedback:                                   │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  1. Overall Assessment                                  │ │
│  │     "Overall, that was a [strong/good/needs work]      │ │
│  │      performance. Here's why..."                       │ │
│  │                                                         │ │
│  │  2. What Went Well (2-3 points)                        │ │
│  │     "I liked how you..."                               │ │
│  │     "Your strength was..."                             │ │
│  │                                                         │ │
│  │  3. Areas for Improvement (2-3 points)                 │ │
│  │     "One area to work on..."                           │ │
│  │     "Next time, consider..."                           │ │
│  │                                                         │ │
│  │  4. Technical Gaps (if any)                            │ │
│  │     "You might want to study..."                       │ │
│  │     "A concept to review is..."                        │ │
│  │                                                         │ │
│  │  5. Hire/No Hire Assessment                            │ │
│  │     "If this were real, I would..."                    │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  Be specific:                                               │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                                                         │ │
│  │  ❌ Vague: "Your database design was weak"             │ │
│  │  ✓ Specific: "You chose SQL but couldn't explain      │ │
│  │     why SQL over NoSQL for this use case. Practice    │ │
│  │     articulating database trade-offs."                │ │
│  │                                                         │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## Evaluation Rubric

```
┌─────────────────────────────────────────────────────────────┐
│                  Evaluation Rubric                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Requirements Gathering (1-5)                            │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  1: Didn't ask questions, made wrong assumptions       │ │
│  │  2: Asked few questions, missed key requirements       │ │
│  │  3: Asked basic questions, understood main needs       │ │
│  │  4: Asked good questions, clear scope definition       │ │
│  │  5: Excellent questions, identified all constraints    │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  2. High-Level Design (1-5)                                 │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  1: No clear architecture, missing components          │ │
│  │  2: Basic architecture, some gaps                      │ │
│  │  3: Reasonable architecture, covers main flow          │ │
│  │  4: Good architecture, well-structured                 │ │
│  │  5: Excellent architecture, optimal choices            │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  3. Technical Depth (1-5)                                   │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  1: Superficial understanding, can't explain choices   │ │
│  │  2: Basic understanding, some gaps                     │ │
│  │  3: Good understanding of core concepts                │ │
│  │  4: Deep understanding, explains trade-offs            │ │
│  │  5: Expert level, discusses internals                  │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  4. Scalability (1-5)                                       │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  1: Doesn't address scale, single points of failure    │ │
│  │  2: Mentions scale but solutions incomplete            │ │
│  │  3: Addresses main scaling concerns                    │ │
│  │  4: Good scaling strategy with justification           │ │
│  │  5: Comprehensive, addresses all bottlenecks           │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  5. Communication (1-5)                                     │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  1: Poor communication, hard to follow                 │ │
│  │  2: Unclear at times, needed prompting                 │ │
│  │  3: Clear communication, drove discussion              │ │
│  │  4: Excellent communication, collaborative             │ │
│  │  5: Outstanding, engaged interviewer effectively       │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  Scoring:                                                   │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  20-25: Strong Hire                                    │ │
│  │  15-19: Hire                                           │ │
│  │  11-14: Lean No Hire                                   │ │
│  │  5-10:  No Hire                                        │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```
